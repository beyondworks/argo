## 변경 안전 절차 — 오류·혼재·버그 0 (2026-07-15 유건 지시, 절대 규칙)

> 사소한 수정·디자인 변경·기능 추가 **모두** 이 절차를 따른다. 셋은 원인이 달라 방어도 다르다 — 변경 크기로 강도만 조절한다.

### 3대 방어선 (실패 유형별)

- **혼재 방지 — 올바른 베이스 + 격리 작업공간.** 작업 시작 전 step 0: "지금 이 코드가 제품 정본인가?"를 확인한다(`git merge-base`·`git log main..HEAD`로 베이스 최신성 점검). 아니면 **올바른 베이스에서 새 브랜치**를 판다(stale 브랜치 위 작업·검수는 있지도 않은 문제를 만든다 — 실측: landing이 제품 보안작업 누락된 stale이었음). **한 변경 = 한 워크트리.** 테스트는 **임시 데이터 루트(임시 `ARGO_ROOT`) + 별도 포트**로 — 실데이터·실행 중 서버(:3001 등) 미접촉. 실험이 실제 상태를 오염시키는 게 혼재의 본질.
- **오류 방지 — 읽고 나서 쓰기 + 최소 변경.** 추측한 줄번호·동작으로 고치지 않는다. 실제 파일을 읽고, **이미 도는 기존 패턴을 재사용**한다(예: 새 화면 로그아웃 = 회사 사이드바의 검증된 `/auth/signout` 폼 재사용). 바꾸는 표면을 최소화. `node --check`(.mjs)·`next build`로 문법·컴파일 확인 — 단 이는 "안 깨짐"이지 "동작"이 아니다.
- **버그 방지 — 수준에 맞는 실제 검증 + 분리 검수.** 빌드 통과 ≠ 동작. UI는 반드시 **띄워서 눈으로**(상태별·라이트/다크·모바일폭). 검수는 만든 컨텍스트가 아닌 **다른 컨텍스트**가 "목적 달성 + 회귀 없음"을 역검증(자기 승인 금지). 같은 계열 갭은 한 번에 안 닫히니 **수정 후 재검증을 반복**한다(실측: 경계 게이트 갭이 3라운드 만에 닫힘).

### 변경 크기별 체크리스트

- **사소(오타·한 줄·설정)**: 정확한 위치 읽기 → 수정 → 문법/빌드. 워크트리·무거운 계획 생략(과설계가 오히려 실수 유발).
- **디자인(UI)**: 수정 → 격리 dev 서버 → **브라우저 시각 확인(상태별)** → 스크린샷 증거. 빌드만으로 완료 선언 금지. 브라우저는 Aside 사용(글로벌 규칙).
- **기능**: 올바른 베이스 → 최소 출하버전(MVP) → 기존 패턴 재사용 → 실제 흐름 실행 → 분리 검수 → 검증 → 논리단위 커밋. 큰 리팩터 겸사겸사·여러 관심사 한 커밋 금지.

### 못 한 검증은 정직하게
자격증명 필요 등으로 라이브 검증 못 한 부분은 **"추론(코드 동일성) — 라이브 아님"**으로 명시한다. "완료/무결/검증 완료" 도장 금지(글로벌 communication 규칙).

### 마감 위생
커밋 전 `git diff --cached`에서 시크릿 평문(key/token/DATABASE_URL) grep. `git add .`/`-A` 금지 — 파일 명시 add. **논리 단위 커밋**(보안/기능/정리 분리, 커밋 간 빌드 유지). 파괴적 행위(삭제·force push·병합)는 **승인 후**.

## 도메인: saas (SaaS)

> 웹/앱 제품 개발, 풀스택, 배포. 사용자 결과 = **동작하는 제품 기능**.

### 표준 파이프라인

탐색(explore) → 계획(plan) → 구현 → 코드/보안 리뷰 → 동작 검증 → 배포

### 추천 에이전트 세트 (1단 기본 라우팅)

| 단계 | 우선 사용 | 비고 |
|---|---|---|
| 설계 | `system-architect`, `backend-architect`, `api-designer` | 구조 먼저 |
| 프론트 | `frontend-architect`, `react-specialist`, `nextjs-developer` | UI 구현 |
| 풀스택 구현 | `fullstack-developer`, `typescript-pro` | 경계 통합 |
| 배포 | `deployment-engineer`, `devops-engineer` | Vercel/systemd |
| 검수(분리) | `code-reviewer` + `security-engineer` + `test-automator` | 자기 승인 금지 |

### 추천 스킬 (작업 종류별)

- 제품 골격: `/saas-platform-builder`, `vercel:bootstrap`, `vercel:nextjs`
- 프론트 디자인: `/frontend-design`, `/ui-ux-pro-max`, `shadcn-ui` MCP
- 규율: `superpowers:test-driven-development`, `superpowers:systematic-debugging`
- 리뷰/배포: `/code-review`, `/verify`, `vercel:deploy`, `vercel:env`

### 도메인 검증 기준 (절대)

- **빌드 성공 ≠ 기능 동작.** UI 변경은 dev 서버 + 브라우저(playwright/ui-inspector) 시각 확인 후 완료.
- 배포 직후 엔드포인트 `curl` 응답 검증.
- 비동기 상태 변경은 렌더링 타이밍까지 확인.
- 시크릿 평문 금지(글로벌 보안 규칙). `.env`는 이름·위치만 기록.

## 다국어 상시 규칙 (2026-07-10 유건 지시 — 위반 금지)

- **모든 UI 문자열은 `app/i18n.jsx`의 사전(t)을 통해서만** 넣는다. 새 기능·페이지 추가 시 지시 없어도 ko/en 두 언어 모두 등록.
- 한국어 모드 = 고유명사(Argo, Claude 등) 제외 전부 한글. 마이크로라벨도 한글.
- 월 지출 한도는 한국어 모드에서 원화(₩, `fmtMoney` 고정 환산율) 표기.
- 언어 전환 단축키 `cmd+/` (i18n Provider가 처리). 언어 상태는 localStorage `argo-lang`.
