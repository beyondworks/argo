# 보안 — 맥락 데이터 암호화 로드맵

> 2026-07-14. 유료 멀티기기 동기화를 팔기 전, "내 맥락 데이터가 우리 Supabase에 저장되면
> 유출·프라이버시 우려가 있다"는 문제를 어떻게 닫는지 정의한다.
> 근거 코드: `src/secretbox.mjs`, `src/sync.mjs`, `supabase/migrations/20260714090000_companies_storage_rls.sql`.

## 현재 상태 (코드 실측 — 숨기지 않는다)

| 데이터 | 처리 | 서버(운영자)·Supabase가 복호 가능? |
|---|---|---|
| 크레덴셜 (`connections.json`, `.secrets.json` = 봇 토큰·러너 키) | **봉투 암호화** v2 (AES-256-GCM, 계정 키 HKDF 파생) | **불가** — 암호문만 저장 |
| **기억·대화·크루 카드·스킬 (= 맥락 데이터)** | **평문 업로드** | 이론상 **가능** |

- 암호화 범위는 `src/secretbox.mjs`의 `isSecretRel(rel) = rel === 'connections.json' || rel === '.secrets.json'` 로 **크레덴셜 2종에 한정**. 그 외 파일은 `src/sync.mjs`에서 평문 업로드.
- 지금 맥락 데이터를 지키는 것: 전송 TLS + Supabase 저장 암호화 + **Storage RLS(본인 경로 `companies/<ownerId>/…`만 접근)**. → 다른 *사용자*는 막지만, 복호 키를 서버가 쥐고 있어 *운영자·Supabase*는 원리상 막지 못한다. 이게 사용자가 우려하는 지점이다.
- **Free(로컬 전용)는 이 우려의 대상이 아니다** — 데이터가 기기를 떠나지 않는다. 이것이 즉시 유효한 최강 답이며, "클라우드 켜기 전엔 한 바이트도 안 나간다"는 사실이자 카피다.

## 마일스톤

### M-ENC-1 — 워크스페이스 전체 봉투 암호화 (유료 동기화 GA 전 **필수**)
크레덴셜에만 쓰는 봉투 암호화(`sealSecret`/`openSecret`, 계정 키 파생)를 **기억·대화·크루·스킬 전체로 확대**한다. 이미 있는 부품의 범위 확장이지 새 발명이 아니다.
- 구현 방향: `isSecretRel` 게이트를 워크스페이스 동기 대상 전반으로 확장(또는 "암호화 대상" 화이트리스트 → 사실상 전체). 계정 키 미확보 사이클의 삭제 오인 가드(`if (isSecretRel(rel) && !cryptoOn()) continue;` 패턴)를 전체 파일로 일반화.
- 효과: 서버는 **계정 키 없이 복호 불가** → 스토리지가 유출돼도 암호문. 로그인-연동 기기는 계정 키로 열람.
- **수용 기준**: (1) 업로드 페이로드가 `argosecret.v2:` 매직으로 시작하는 암호문임을 스토리지에서 실측, (2) 새 기기 로그인 후 기억·대화가 정상 복호·렌더, (3) 스토리지에 평문 맥락 파일 **0건**, (4) 계정 키 회전 시 재봉인 동작.
- 한계: 계정 키를 서버가 파생·보관하는 구조상 완전한 제로지식은 아님(그건 M-ENC-2). 하지만 "유출돼도 암호문"의 실질 방어선을 GA 전에 세운다.

### M-ENC-2 — 진짜 E2E / 제로지식 (= 기존 M-3 "진짜 E2E")
복호 키를 **사용자만** 보유(패스프레이즈 파생 키 또는 기기 보유 키). 서버는 암호문만 보관하고 복호 키를 **절대 갖지 않는다** — Obsidian Sync·1Password 모델.
- **수용 기준**: (1) 서버 로그·DB·스토리지 어디에도 평문·복호 키 부재를 실측, (2) 키 없이 복구 불가 확인(제로지식 증명), (3) 비밀번호 분실 = 복구 불가 트레이드오프에 대한 명시적 UX(경고·복구 코드 등).
- 트레이드오프: 분실 복구 불가·검색/서버측 처리 제약 → UX 설계가 관건. 그래서 M-ENC-1(운영 방어) 다음의 신뢰 강화 단계.

### 지원 레이어 (병행 — 기술 + 사회적 증거)
- **데이터 소유권**: 원클릭 내보내기/완전 삭제.
- **자가 호스팅**: 민감 기업은 자기 Supabase/인프라에 연결(`SUPABASE_SERVICE_ROLE_KEY`를 자기 것으로). 코드가 이미 서비스키 기반 self-host 구조를 가짐 — SaaS 멀티테넌트 출하 시 테넌트 스코프 자격 분리 필요.
- **리전 선택 · 외부 감사(SOC2 등)**: 규모 성장 시.

## GA 게이트 (결정)

- **M-ENC-1을 유료 멀티기기 동기화 GA 전 필수 관문으로 둔다.** 통과 전에는 "클라우드 동기화"를 파는 대신 로컬-우선(Free)만 전면에 세운다.
- Free 로컬 사용에는 암호화 확장이 무관(데이터가 안 나감) — 게이트는 클라우드 동기화에만 적용.
- M-ENC-2(E2E)는 결제 뒤 신뢰 강화 단계(기존 M-3 순서 유지).

## 검증 원칙
- 각 수용 기준은 **스토리지·서버 로그 실측**으로 확인(코드 리뷰만으로 "안전" 선언 금지).
- 시크릿 평문 금지(글로벌 보안 규칙) — 이 문서에도 env는 이름만.
